1. Project Overview
The IPL Auction Management System is a real-time, web-based platform designed to digitize and streamline the management of mock IPL (Indian Premier League) player auctions for college events. The system provides a controlled, two-role environment: an Admin who manages the auction in real time, and a Viewer (audience or participant) who follows live updates on a read-only scoreboard.

1.1  Problem Statement
Manual IPL auction events at colleges suffer from transparency issues, calculation errors, slow purse tracking, and an inability to show live updates to large audiences. This system eliminates those gaps with a structured digital platform.

1.2  Objectives
Provide a secure admin interface to manage team purses in real time.
Provide a public viewer board showing live purse standings.
Broadcast all auction events instantly via WebSockets.
Prevent invalid transactions with robust input validation.
Deliver an IPL-themed, visually engaging UI suitable for projector display.

4. Non-Functional Requirements
4.1  Performance
WebSocket event delivery latency: < 500 ms under normal network conditions.
Page initial load time: < 2 seconds on a standard college Wi-Fi network.
API response time for /deduct: < 300 ms (excluding network latency).
System must handle at least 100 concurrent viewer WebSocket connections.

4.2  Reliability & Availability
Uptime target: 99% during the event window (typically 3–6 hours).
Firebase/Firestore provides automatic data persistence; no data loss on page refresh.
Graceful reconnection: WebSocket clients auto-reconnect within 3 seconds on drop.

4.3  Security
Admin login endpoint rate-limited to 5 attempts per minute per IP.
JWT tokens expire after 8 hours; refresh not required for event duration.
HTTPS enforced in production deployment.
All input sanitized on the server before DB writes (prevent injection).
Environment variables used for all secrets (JWT_SECRET, DB credentials).

4.4  Usability
UI must be operable by a non-technical event volunteer after 5-minute briefing.
Error messages must be human-readable and actionable.
Viewer board must be self-explanatory with no documentation required.

4.5  Responsiveness
Admin panel: optimized for laptop/desktop (1280px+ width).
Viewer board: responsive grid from 360px (mobile) to 1920px (projector/TV).

5. System Architecture
5.1  High-Level Architecture
The system follows a three-tier architecture: React frontend, Node.js/Express backend, and Firebase Realtime Database. Socket.io overlays the HTTP server to maintain a persistent duplex channel for real-time event broadcasting.

5.2  Data Flow — Deduction Event
Admin enters bid amount and clicks Deduct.
React frontend POSTs to /api/deduct with JWT in Authorization header.
Express middleware validates JWT; rejects 401 if invalid.
Controller validates: amount > 0, amount <= current purse, team exists.
On success: Firebase updates team purse; server emits purse_update via Socket.io.
All connected viewer clients receive the event and re-render purse value.
Admin UI updates immediately (optimistic or on confirmation).

13. Deployment & Operations
13.1  Deployment Architecture
Frontend (React) deployed to Vercel or Netlify — static CDN hosting, free tier.
Backend (Node.js + Socket.io) deployed to Render or Railway — persistent Node.js instance required for WebSocket support.
Firebase Realtime Database hosted on Google's infrastructure — no self-hosting.
Custom domain optional; HTTPS automatically provisioned by Vercel/Render.

13.3  Pre-Event Checklist
Seed Firebase with all 10 team records and starting purse values.
Verify admin login works on the deployed URL.
Open viewer board on projector browser in full-screen mode.
Test one deduction from admin; confirm viewer board updates live.
Ensure both admin and viewer devices are on the same Wi-Fi network (or internet).
Keep a backup admin device (tablet/laptop) with the admin URL bookmarked.
Test reset functionality before event starts.

15. Glossary
Term
Definition
Purse
The remaining monetary budget allocated to a team for buying players in the auction
Deduction
The act of subtracting a bid amount from a team's purse after a player is sold to them
JWT
JSON Web Token — a compact, self-contained method for securely transmitting session information
WebSocket
A persistent bidirectional communication protocol enabling real-time data push from server to clients
Socket.io
A JavaScript library built on WebSockets with automatic fallback and reconnection support
Firebase RTDB
Firebase Realtime Database — a cloud-hosted NoSQL JSON database with real-time sync
REST API
Representational State Transfer API — HTTP-based endpoints for CRUD operations
Admin Panel
The password-protected interface used by the auction controller to manage purses
Viewer Board
The public, read-only scoreboard displayed to the audience during the auction
